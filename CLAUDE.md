# CLAUDE.md

## 1. Test First, Always

**所有代码变更必须从测试开始。没有例外。**

- Bug fix → 先写一个能复现 bug 的失败测试，再修复
- New feature → 先写定义预期行为的测试，再实现
- Refactor → 先确认现有测试通过，补充不足的测试，再重构

严格遵循 Red → Green → Refactor 循环。永远不要跳过 Red。

如果你即将写实现代码，但还没有对应的失败测试 — 停下来，先写测试。

## 2. Think Before Coding

**不要假设。不要隐藏困惑。遵循已有流程。**

开始实现之前：
- 明确说出你的假设。不确定的就问。
- 如果有多种理解方式，列出来 — 不要默默选一个。
- 当有已建立的工作流或 skill 时，使用它们。不要跳过流程。
- 如果有更简单的方案，说出来。该 push back 就 push back。

## 3. Simplicity First

**最少的代码解决问题。不做推测性设计。**

- 不加需求之外的功能。
- 不为一次性代码创建抽象。
- 不加未被要求的"灵活性"或"可配置性"。
- 不为不可能的场景做错误处理。
- 200 行能用 50 行解决的，重写。

自问："一个 senior engineer 会觉得这过度设计了吗？" 如果是，简化。

## 4. Surgical Changes

**只改必须改的。只清理你自己造成的问题。**

编辑已有代码时：
- 不要"顺便改进"周围的代码、注释或格式。
- 匹配已有风格，即使你会用不同的方式。
- 清除你的改动导致的无用 import/变量/函数。
- 发现不相关的问题，提出来 — 不要动手修。

检验标准：每一行变更都应能直接追溯到用户的请求。

## 5. Execute Fully, Decide Together

**执行权交给你，决策权留给我。**

- 能用工具完成的执行类任务（读代码、写实现、跑测试、查资料、修小问题）→ 自主完成，不要每步都征求许可。
- 需要权衡或不可逆的决策（架构选型、范围调整、删数据、推送代码、改共享系统）→ 必须我来拍板。
- 边界判断：可逆 + 影响小 = 执行；不可逆 或 影响大 = 决策。
- 中途遇到障碍，先用合理方案推进；真正卡住或需要权衡时再问。
- 提问方式：需要我决策、或要澄清第 2 条里的假设时，**统一使用 `AskUserQuestion` 工具**，不要在普通文本里夹带问题让我手动回复。

具体例子（这些都属于执行，自己做完，不要让我做）：
- 开发完需求后重启 dev server / 后端服务 — 自己重启并验证，不要提醒"请重启服务"。
- 跑测试、装依赖、清缓存、生成 migration、跑 lint/format — 自己执行。
- 查日志、查报错、查文档、查 API 用法 — 自己查，不要让我贴给你。

检验标准：我的介入次数应匹配决策点，而不是工具调用次数。

---

**这些原则在起作用的标志：** 测试先于实现代码出现，diff 最小化，澄清问题在动手之前而非出错之后，介入只发生在决策点。
